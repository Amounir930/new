
Store-#01 Home Page 🚀
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُعد الصفحة الرئيسية الواجهة الأساسية للمتجر، وتهدف إلى تقديم تجربة مستخدم سريعة، جذابة، ومُحسّنة لمحركات البحث (SEO). يتم بناء الصفحة باستخدام Next.js للاستفادة من مزايا العرض من جانب الخادم (SSR) والتجديد الثابت المتزايد (ISR).

بنية الصفحة:
النقطة الرئيسية: تحديد مسار الصفحة وإنشاء المكون الأساسي.
كيفية التنفيذ: إنشاء ملف src/apps/storefront/app/[tenantSlug]/page.tsx في بيئة Next.js. هذا المسار الديناميكي يسمح لكل tenant بالحصول على صفحته الرئيسية الخاصة.
عرض المحتوى والأداء:
النقطة الرئيسية: ضمان التحميل السريع للمحتوى وتحديثه بانتظام.
كيفية التنفيذ:
Server-Side Rendering (SSR): تُستخدم SSR للتحميل الأولي للصفحة، مما يضمن عرض المحتوى بسرعة للمستخدمين ولبرامج زحف محركات البحث، وهو أمر حيوي لتحسين SEO.
Incremental Static Regeneration (ISR): يتم تفعيل ISR عن طريق تعيين revalidate لمدة زمنية محددة (مثل 60 ثانية). هذا يسمح بإعادة إنشاء الصفحة في الخلفية بانتظام (مثال: كل دقيقة) لجلب أي تحديثات للمحتوى دون الحاجة لإعادة نشر التطبيق بالكامل.
مصدر البيانات:
النقطة الرئيسية: جلب المحتوى الديناميكي للصفحة الرئيسية من مصادر قابلة للإدارة.
كيفية التنفيذ: يتم جلب أقسام الصفحة الرئيسية (مثل Hero Banners، المنتجات المميزة، الأكثر مبيعًا) من API endpoint مخصص: GET /api/storefront/home-sections?tenantId={}. يتم تكوين هذه الأقسام بواسطة المسؤولين من خلال لوحة التحكم (Admin-#04) وتُخزن في قاعدة البيانات ضمن tenant_config أو content_sections table.
التخزين المؤقت المتقدم (Advanced Caching):
النقطة الرئيسية: تحسين زمن استجابة API وتقليل حمل قاعدة البيانات.
كيفية التنفيذ: استخدام Redis Cache (Arch-Core-07) لتخزين استجابات API الخاصة بأقسام الصفحة الرئيسية. يتم تطبيق آليات Cache invalidation تلقائية لضمان تحديث الكاش عند تعديل المحتوى من لوحة التحكم (Admin Panel).
تحسين الأداء (Performance Optimization):
النقطة الرئيسية: تقليل أوقات التحميل وتحسين تجربة المستخدم.
كيفية التنفيذ:
next/image: لتحميل الصور بشكل كسول (Lazy Loading) وتوليد صور متجاوبة (responsive images) بأحجام مختلفة حسب الجهاز.
next/font: لتحسين تحميل الخطوط وتقليل أثرها على أداء الصفحة.
Code Splitting: لتقسيم حزم الكود (JS bundle) إلى أجزاء أصغر، مما يقلل من الوقت اللازم لتحميل وتحليل الكود الأولي.
تحسين محركات البحث (SEO):
النقطة الرئيسية: التأكد من قابلية اكتشاف الصفحة وتصنيفها جيدًا في محركات البحث.
كيفية التنفيذ: تضمين Meta Tags الأساسية (title, description) و JSON-LD Schema Markup للمنتجات والمنظمات، مما يساعد محركات البحث على فهم محتوى الصفحة بشكل أفضل.
المعلومات البرمجية التقنية:

@apex/db (Drizzle ORM): للوصول إلى قاعدة البيانات (PostgreSQL) وجلب بيانات tenant_config و content_sections.
Redis Cache (Arch-Core-07): لتخزين استجابات API مؤقتًا وتحسين الأداء.
Next.js (Frontend Framework): لبناء الواجهة الأمامية، وإدارة التوجيه (Routing)، وتوفير SSR وISR.
Admin-#04 (Content Configuration): توفر واجهة لإدارة وتكوين محتوى الصفحة الرئيسية الديناميكي.
ملاحظات:

استخدام ISR مع revalidate يوازن بين تحديث المحتوى الفوري وأداء الصفحات الثابتة.
Cache invalidation يجب أن تكون آلية وموثوقة لضمان عرض المحتوى الأحدث دائمًا بعد تحديثه من Admin Panel.
التركيز على Core Web Vitals (LCP, FID, CLS) ضروري لتحقيق درجات Lighthouse عالية وتجربة مستخدم ممتازة.
اختبارات القبول:

أداء Lighthouse: تحقيق درجة أداء > 90 على Google Lighthouse لـ Performance و SEO على الأجهزة المحمولة والكمبيوتر.
تحديث المحتوى: أي تغيير في المحتوى الديناميكي للصفحة الرئيسية من Admin Panel ينعكس على Storefront في غضون 60 ثانية كحد أقصى (فترة ISR).
Cache Hit Ratio: مراقبة Redis Cache تظهر cache hit ratio > 95% لطلبات بيانات الصفحة الرئيسية.
زمن الاستجابة (TTFB): Time To First Byte (TTFB) أقل من 200ms للتحميل الأولي للصفحة.
تجاوب كامل: الصفحة تعرض بشكل مثالي على جميع الأجهزة (Mobile, Tablet, Desktop) وتعمل بشكل سلس مع سرعات اتصال مختلفة.
Store-#03 Product Details (PDP) 🚀
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

صفحة تفاصيل المنتج (PDP) هي صفحة حاسمة لتحويل الزوار إلى عملاء. تركز هذه الصفحة على عرض معلومات المنتج بشكل شامل وجذاب، مع توفير تجربة سريعة ومتجاوبة.

بنية الصفحة:
النقطة الرئيسية: تحديد مسار الصفحة وإنشاء المكون الخاص بها.
كيفية التنفيذ: إنشاء ملف src/apps/storefront/app/[tenantSlug]/products/[slug]/page.tsx في Next.js، حيث [slug] يمثل المعرف الفريد للمنتج.
جلب البيانات:
النقطة الرئيسية: تجميع جميع معلومات المنتج ذات الصلة من مصادر متعددة.
كيفية التنفيذ:
جلب تفاصيل المنتج الأساسية، المتغيرات (variants)، المخزون (inventory)، ومراجعات العملاء من API endpoint مخصص: GET /api/storefront/products/{}?withReviews=true.
استخدام pgvector (PostgreSQL extension) على الـ Backend لإجراء بحث تشابه (similarity search) قائم على التضمين (embeddings) للمنتجات ذات الصلة (Related Products). يتم تحديث هذه التضمينات بشكل غير متزامن عبر BullMQ عند تحديث المنتجات.
إدارة حالة السلة:
النقطة الرئيسية: توفير تفاعل فوري عند إضافة المنتجات إلى السلة.
كيفية التنفيذ: زر "Add to Cart" يتفاعل مع Zustand store (الموجود في packages/storefront-state) لإدارة حالة السلة من جانب العميل. يتم تحديث أيقونة السلة (badge) فورًا لتعكس العدد الجديد من العناصر.
تحسين الصور:
النقطة الرئيسية: ضمان تحميل الصور بسرعة وبجودة عالية، مع التكيف مع أحجام الشاشات المختلفة.
كيفية التنفيذ: يتم تقديم جميع صور المنتج (الصور الرئيسية، المصغرة، صور المتغيرات) عبر Imgproxy (Arch-Core-09). Imgproxy مسؤول عن:
التحويل إلى صيغة WebP لتقليل حجم الملفات.
تغيير الحجم (resizing) ديناميكيًا بناءً على حجم شاشة المستخدم.
تحسين الجودة تلقائيًا.
تجربة المستخدم (UX):
النقطة الرئيسية: تحسين عرض المحتوى وتفاعل المستخدم مع الصفحة.
كيفية التنفيذ:
Lazy Loading: لصور المنتج الثانوية لتسريع التحميل الأولي.
مكونات Accordion: لعرض المواصفات والوصف بطريقة منظمة وموفرة للمساحة.
مكونات Carousel: لعرض الصور المتعددة للمنتج بشكل جذاب وتفاعلي.
المعلومات البرمجية التقنية:

Zustand 🐻 (Client-side State Management): لإدارة حالة سلة التسوق بشكل فعال وسريع على جانب العميل.
pgvector (PostgreSQL Vector Search): لتفعيل البحث الدلالي عن المنتجات ذات الصلة بناءً على التضمينات.
Imgproxy (Arch-Core-09): لمعالجة الصور وتحسينها ديناميكيًا.
Next.js (Frontend Framework): لبناء الواجهة الأمامية وإدارة مسارات الصفحة.
@apex/db (Drizzle ORM): للتفاعل مع قاعدة البيانات وجلب تفاصيل المنتجات.
ملاحظات:

استخدام pgvector مع التضمينات يُعزز بشكل كبير دقة قسم "المنتجات ذات الصلة"، مما يزيد من فرص البيع العابر.
التحسين المستمر للصور عبر Imgproxy أمر بالغ الأهمية للحفاظ على أداء عالٍ للصفحة.
تحديث حالة السلة الفوري عبر Zustand يساهم في تجربة مستخدم سلسة وخالية من التأخير.
اختبارات القبول:

جودة الصور: جميع صور المنتج على صفحة PDP يتم تحميلها كصور WebP محسّنة (التحقق من خلال Network tab في أدوات المطور بالمتصفح).
وظائف السلة: النقر على "Add to Cart" يؤدي إلى زيادة عدد العناصر في أيقونة السلة فورًا دون تحديث الصفحة.
منتجات ذات صلة: قسم "Related Products" يعرض 3-5 منتجات ذات صلة، ويتم تحميلها في أقل من 200ms.
أداء الصفحة: صفحة PDP بالكامل (مع الصور والمحتوى) يتم تحميلها وعرضها بشكل كامل (LCP - Largest Contentful Paint) في أقل من 2 ثانية.
تجاوب: الصفحة تعمل بشكل سلس ومتجاوب على مختلف أحجام الشاشات.
Store-#13 Login Modal ⚡
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

نموذج تسجيل الدخول المنبثق يوفر طريقة سريعة ومريحة للمستخدمين لتسجيل الدخول دون مغادرة الصفحة الحالية، مع التركيز على الأمان وسهولة الاستخدام.

بنية المكون:
النقطة الرئيسية: استخدام مكون جاهز لواجهة المستخدم لضمان الاتساق وسهولة الوصول.
كيفية التنفيذ: استخدام مكون Dialog من Radix UI (مثال: packages/ui/radix/dialog) لإنشاء نافذة تسجيل الدخول المنبثقة. هذا المكون يضمن السلوك الصحيح، والوصولية (accessibility)، ويمكن استدعاؤه من أي مكان في Storefront.
التحقق من الصحة (Validation):
النقطة الرئيسية: التحقق من صحة بيانات الإدخال لضمان الدقة والأمان.
كيفية التنفيذ:
Zod: استخدام Zod لتعريف Schema لبيانات نموذج تسجيل الدخول (email, password).
Client-side Validation: تطبيق التحقق من الصحة على جانب العميل لتقديم ملاحظات فورية للمستخدم وتحسين تجربة المستخدم.
Server-side Validation: تطبيق التحقق من الصحة على جانب الخادم عبر NestJS Pipes (Arch-Core-05) لضمان أمان البيانات قبل معالجتها.
المصادقة (JWT Authentication):
النقطة الرئيسية: توفير آلية آمنة للمصادقة وتخزين جلسة المستخدم.
كيفية التنفيذ: عند إرسال بيانات اعتماد صحيحة، يقوم الـ Backend (ضمن @apex/auth module) بإصدار JWT (JSON Web Token) موقع بشكل آمن. يتم تخزين هذا الـ JWT في httpOnly, secure, SameSite=Lax cookie لمنع الوصول إليه من خلال JavaScript الضار.
تسجيل الدخول الاجتماعي (Social Login):
النقطة الرئيسية: توفير خيارات تسجيل دخول بديلة ومريحة.
كيفية التنفيذ: دمج NextAuth.js باستخدام موفري الهوية (providers) مثل Google، Apple، و Facebook. يتولى NextAuth.js إدارة تدفقات OAuth وتخزين بيانات المستخدم بشكل آمن.
أمان كلمة المرور:
النقطة الرئيسية: حماية كلمات مرور المستخدمين المخزنة في قاعدة البيانات.
كيفية التنفيذ: تخزين كلمات المرور في قاعدة البيانات يتم بعد تجزئتها (hashing) باستخدام bcrypt مع عدد جولات (rounds) كافٍ (مثال: 12-14). هذا يمنع هجمات القاموس (dictionary attacks) وقوة التكسير (brute-force attacks).
المعلومات البرمجية التقنية:

@apex/auth (Authentication Module): مسؤول عن منطق المصادقة وإصدار JWT.
Zod (Validation): لتعريف Schemas والتحقق من صحة بيانات الإدخال.
bcrypt (Password Hashing): لتجزئة (hashing) كلمات المرور بأمان.
NextAuth.js (Social Login Integration): لتبسيط عملية دمج تسجيل الدخول الاجتماعي.
Radix UI (UI Components): لتوفير مكون Dialog المنبثق.
Arch-Core-05 (Global Input Validation): توفير بنية تحتية للتحقق من صحة الإدخال على جانب الخادم.
ملاحظات:

استخدام httpOnly و secure و SameSite=Lax للكوكيز التي تحتوي على JWT أمر بالغ الأهمية للأمان ويحمي من هجمات XSS و CSRF.
يجب أن تكون رسائل الخطأ عامة قدر الإمكان لتجنب تسريب معلومات قد يستغلها المهاجمون (مثال: "البريد الإلكتروني غير موجود" بدلاً من "كلمة المرور خاطئة").
مراجعة دورية لإعدادات أمان NextAuth.js و bcrypt ضرورية.
اختبارات القبول:

تسجيل دخول ناجح: إدخال بيانات اعتماد صالحة يؤدي إلى تسجيل الدخول بنجاح وإعادة توجيه المستخدم إلى صفحة ما بعد تسجيل الدخول (مثال: /account).
رسائل خطأ دقيقة: عند فشل تسجيل الدخول (مثال: كلمة مرور خاطئة، بريد إلكتروني غير موجود)، يتم عرض رسالة خطأ محددة وواضحة للمستخدم.
أمان الـ JWT: لا يمكن لـ JavaScript من جانب العميل الوصول إلى JWT المخزن في الـ httpOnly cookie.
وظائف تسجيل الدخول الاجتماعي: تسجيل الدخول عبر Google يعمل بشكل صحيح، ويتم إنشاء حساب للمستخدم أو ربطه بحساب موجود.
تقارير أمان: اختبارات أمان (SAST, DAST) لا تكتشف ثغرات شائعة (مثل XSS, CSRF) في مكون تسجيل الدخول.
Store-#06 Checkout (One-Page) ⚡🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

عملية الدفع من صفحة واحدة تهدف إلى تبسيط تجربة الشراء وتقليل معدلات التخلي عن السلة، مع ضمان الأمان والامتثال.

بنية النموذج:
النقطة الرئيسية: توفير تدفق دفع متعدد الخطوات على صفحة واحدة.
كيفية التنفيذ: تطوير نموذج دفع من صفحة واحدة متعدد الخطوات (Shipping Address → Shipping Method → Payment Information). يتم استخدام Next.js Router للتبديل بين الخطوات دون الحاجة لإعادة تحميل الصفحة بالكامل، مما يوفر تجربة سريعة وسلسة.
معالجة الدفع (Stripe):
النقطة الرئيسية: دمج بوابة دفع آمنة ومرنة.
كيفية التنفيذ:
Stripe Elements JS SDK: دمج حقول إدخال بطاقة الائتمان المتوافقة مع PCI-DSS (مثال: CardElement, PaymentElement) من Stripe مباشرة في الواجهة الأمامية. هذا يضمن أن معلومات البطاقة الحساسة لا تلامس خوادمنا أبدًا.
PaymentIntent: يتم إرسال PaymentIntent إلى الـ Backend، والذي يتفاعل مع Stripe API لمعالجة الدفع.
التحقق من الصحة (Validation):
النقطة الرئيسية: ضمان صحة جميع البيانات المدخلة في نموذج الدفع.
كيفية التنفيذ: يتم التحقق من صحة جميع حقول النموذج (العنوان، معلومات الشحن) باستخدام Zod على جانبي العميل والخادم (عبر Arch-Core-05).
تتبع حالة الطلب:
النقطة الرئيسية: إدارة دورة حياة الطلب وتحديث حالته بشكل موثوق.
كيفية التنفيذ:
إنشاء الطلب: بعد معالجة الدفع بنجاح، يتم إنشاء طلب في قاعدة البيانات (PostgreSQL) بحالة "Pending Payment".
Stripe Webhook: يتم تكوين Stripe Webhook (موجه إلى API endpoint آمن لدينا). عند تلقي إشعار payment_intent.succeeded من Stripe، يتم تحديث حالة الطلب إلى "Paid".
إدارة معلومات التعريف الشخصية (PII):
النقطة الرئيسية: حماية أي بيانات دفع حساسة إضافية.
كيفية التنفيذ: إذا تطلبت أي بيانات دفع حساسة (بخلاف ما يتعامل معه Stripe مباشرة) التخزين، فسيتم تشفيرها باستخدام خدمة التشفير (Arch-S7).
إشعارات العملاء:
النقطة الرئيسية: إبلاغ العملاء بحالة طلباتهم.
كيفية التنفيذ: بعد تأكيد الطلب، يتم إرسال مهمة BullMQ (Arch-Core-07) لإرسال تأكيد الطلب عبر البريد الإلكتروني إلى العميل.
المعلومات البرمجية التقنية:

Stripe SDK (Payment Gateway): لتكامل معالجة الدفع الآمنة.
Zod (Validation): لتعريف Schemas والتحقق من صحة بيانات الإدخال.
S7 Encryption (Arch-S7): لتشفير أي بيانات حساسة إضافية تتطلب التخزين.
BullMQ (Arch-Core-07): لإدارة المهام غير المتزامنة مثل إرسال الإشعارات.
Next.js (Frontend Framework): لبناء الواجهة الأمامية وإدارة التوجيه.
Arch-Core-05 (Global Input Validation): لتوفير التحقق من الصحة على جانب الخادم.
ملاحظات:

الامتثال لمعيار PCI-DSS أمر بالغ الأهمية، واستخدام Stripe Elements يقلل بشكل كبير من عبء الامتثال على التطبيق.
معالجة الـ Webhooks من Stripe يجب أن تكون قوية وتتعامل مع حالات الفشل المحتملة والتكرار (idempotency).
يجب أن تكون خدمة التشفير (Arch-S7) مستندة إلى أفضل الممارسات في التشفير لتأمين PII.
اختبارات القبول:

عملية الدفع كاملة: يتمكن المستخدم من إكمال عملية دفع تجريبية بنجاح باستخدام بطاقة اختبارية.
تحديث حالة الطلب: يتم إنشاء الطلب في قاعدة البيانات بحالة "Pending Payment" ثم يتم تحديثه إلى "Paid" فورًا بعد معالجة Stripe Webhook.
رسائل خطأ دقيقة: عند فشل الدفع (مثال: بطاقة مرفوضة)، يتم عرض رسالة خطأ واضحة ومفيدة للمستخدم.
أمان PCI-DSS: فحص أمان خارجي يؤكد أن تطبيقنا لا يخزن أو يعالج بيانات بطاقة الائتمان الحساسة مباشرة.

كيفية التنفيذ: تصميم وتنفيذ اللافتة والميكانيكية لتكون متوافقة مع لوائح حماية البيانات مثل GDPR و CCPA.

كيفية التنفيذ: يتم عرض اللافتة فقط في الزيارة الأولى للمستخدم أو إذا لم يتم تخزين تفضيل الموافقة في localStorage.

ملاحظات:

الامتثال للوائح مثل GDPR و CCPA أمر بالغ الأهمية لتجنب العقوبات وبناء ثقة المستخدم.

Admin-#21 Bulk Import/Export
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

واجهة المستخدم:

عملية الاستيراد (Import):

كيفية التنفيذ:
رفع الملف: يسمح للمستخدم برفع ملف CSV.
تحليل CSV: يتم استخدام PapaParse (على جانب العميل) لتحليل ملف CSV بسرعة.
إرسال البيانات: يتم إرسال البيانات المحللة (بجزء صغير إذا كانت كبيرة جدًا) إلى API endpoint (مثال: POST /admin/products/import).
التحقق من الصحة والعمليات الخلفية: الـ Backend يقوم بـ Validation صارم للبيانات باستخدام Zod (Arch-Core-05). ثم يتم إرسال مهمة BullMQ (Arch-Core-07) لمعالجة الاستيراد في الخلفية (batch insertion عبر Drizzle ORM).
تحديث حالة المهمة: تزويد المستخدم بتحديثات حالة المهمة في UI.
عملية التصدير (Export):
النقطة الرئيسية: تصدير المنتجات إلى ملف CSV بشكل موثوق.
كيفية التنفيذ:
تشغيل المهمة: عند طلب التصدير، يتم تشغيل مهمة BullMQ (Arch-Core-07) لإنشاء ملف CSV من جميع المنتجات.
تخزين الملف: يتم تخزين ملف CSV الناتج في MinIO (Arch-Core-02) (مثال: في مجلد exports/).
تنزيل آمن: يتم تزويد المستخدم بـ Signed MinIO URL (Arch-Core-09) لتنزيل الملف بشكل آمن ومؤقت.
المعلومات البرمجية التقنية:

Drizzle ORM: للتفاعل مع قاعدة البيانات لإضافة/تحديث/جلب المنتجات.
Zod (Arch-Core-05): للتحقق من صحة بيانات المنتجات المستوردة.
BullMQ (Arch-Core-07): لإدارة مهام الاستيراد والتصدير في الخلفية.
PapaParse (CSV Parsing Library): لتحليل ملفات CSV على جانب العميل.

Admin-#27 Staff RBAC ⚡
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

نظام التحكم في الوصول المستند إلى الأدوار (RBAC) للموظفين يضمن أن كل موظف لديه الصلاحيات اللازمة لمهامه فقط، مع تعزيز الأمان وعزل البيانات.

نموذج البيانات:
النقطة الرئيسية: تصميم قاعدة بيانات مرنة لتحديد الأدوار والصلاحيات.
كيفية التنفيذ: تصميم مصفوفة صلاحيات مفصلة في قاعدة البيانات (PostgreSQL) باستخدام الجداول التالية:
staff_roles: الأدوار المتاحة (مثال: Admin, Editor, Viewer).
permissions: الصلاحيات المتاحة (مثال: product:create, product:edit, order:view, order:refund, settings:manage).
role_permissions: جدول ربط يحدد الصلاحيات لكل دور.
staff_users: لربط الموظفين بالأدوار.
NestJS Guards:
النقطة الرئيسية: تطبيق التحكم في الوصول على مستوى API.
كيفية التنفيذ: تطوير NestJS Guards مخصصة (مثال: PermissionsGuard) يتم تطبيقها على مسارات الـ API (endpoints) للتحقق من صلاحيات المستخدم قبل السماح بالوصول. الـ Guard يستخلص دور المستخدم من JWT (بعد المصادقة) ويقارنه بالصلاحيات المطلوبة للمسار.
تحديد نطاق Tenant:
النقطة الرئيسية: ضمان أن صلاحيات الموظف تنطبق فقط على Tenant الخاص به.
كيفية التنفيذ: استخدام TenantScopedGuard (Arch-Core-04) لضمان أن صلاحيات الموظف تنطبق فقط على الـ Tenant الذي يعمل فيه، ولا يمكن للموظف في tenant-a إجراء عمليات في tenant-b.
واجهة المستخدم:
النقطة الرئيسية: توفير تجربة مستخدم واضحة تعكس صلاحيات الموظف.
كيفية التنفيذ: في Admin Panel، يتم إخفاء أو تعطيل عناصر واجهة المستخدم (أزرار، حقول إدخال) التي لا يملك الموظف صلاحية استخدامها، مما يوفر تجربة مستخدم واضحة ويمنع الأخطاء.
إدارة الأدوار:
النقطة الرئيسية: توفير آلية لإدارة الأدوار والصلاحيات.
كيفية التنفيذ: Super Admin (في Epic 3) هو المسؤول عن تعريف الأدوار، وتعيين الصلاحيات لهذه الأدوار، وإدارة الموظفين وتعيين أدوار لهم.
المعلومات البرمجية التقنية:

TenantScopedGuard (Arch-Core-04): لفرض عزل البيانات على مستوى Tenant.
JWT (with roles/permissions claims): لتخزين معلومات الدور والصلاحيات في الرمز المميز بعد المصادقة.
NestJS Guards (for API): لتطبيق منطق التحكم في الوصول على مسارات API.
Drizzle ORM (for DB schema): لتعريف والتفاعل مع Schemas قواعد البيانات للأدوار والصلاحيات.
ملاحظات:

تصميم جيد لنموذج RBAC يوازن بين المرونة والأمان وسهولة الإدارة.
يجب أن تكون Guards شاملة ومطبقة على جميع المسارات الحساسة في الـ Backend.
إخفاء عناصر واجهة المستخدم يعزز الأمان ويقلل من الالتباس للموظفين.
اختبارات القبول:

تحكم بالوصول (API): موظف بدور "Viewer" يحاول الوصول إلى POST /admin/products → يتم رفض طلبه بـ 403 Forbidden.
تحكم بالوصول (UI): في واجهة المستخدم، موظف بدور "Viewer" لا يرى زر "Add New Product" أو زر "Delete Order".
عزل Tenant: موظف في tenant-a لا يمكنه رؤية أو تعديل بيانات في tenant-b، حتى لو كان لديه نفس الصلاحيات.
اختبارات أمان: اختبارات أمان (مثل Pen-testing) لا تكتشف أي تجاوزات لنظام RBAC.
توثيق RBAC: وجود توثيق واضح لمصفوفة الأدوار والصلاحيات.

النقطة الرئيسية: حفظ إعدادات الهوية في قاعدة البيانات.
كيفية التنفيذ: يتم تخزين جميع إعدادات الهوية هذه (بما في ذلك روابط الصور) في جدول tenant_config في قاعدة البيانات (PostgreSQL).
نشر التغييرات في الوقت الفعلي:
النقطة الرئيسية: تحديث واجهة Storefront فورًا بعد حفظ التغييرات.
كيفية التنفيذ:
Redis Pub/Sub: عند حفظ التغييرات في Admin Panel، يتم إرسال حدث عبر Redis Pub/Sub (Arch-Core-07) (مثال: tenant_config_updated:{tenantId}).
تحديث Storefront: مثيلات Storefront تستمع إلى هذه الأحداث وتقوم بإعادة جلب إعدادات tenant_config الخاصة بها، مما يؤدي إلى تحديث الشعار والألوان في الواجهة الأمامية بشكل فوري أو شبه فوري.
تحسين الصور:
النقطة الرئيسية: ضمان تحميل الصور المحسّنة في Storefront.
كيفية التنفيذ: يتم خدمة الشعار والأيقونة عبر Imgproxy (Arch-Core-09) لضمان تحسينها (التحويل إلى WebP، تغيير الحجم ديناميكيًا) لأداء Storefront.
المعلومات البرمجية التقنية:

MinIO (Arch-Core-02): لتخزين ملفات الصور (الشعار، الأيقونة).
Redis Pub/Sub (Arch-Core-07): لنشر تحديثات التكوين في الوقت الفعلي.
Imgproxy (Arch-Core-09): لمعالجة الصور وتحسينها.
Drizzle ORM (Database access): للتفاعل مع جدول tenant_config.
S2 Isolation (Arch-Core-04): لضمان أن الإعدادات خاصة بكل Tenant.
ملاحظات:

نظام Pub/Sub عبر Redis ضروري لتحقيق تحديثات فورية للإعدادات عبر جميع مثيلات Storefront.
التحسين التلقائي للصور عبر Imgproxy يضمن أداء تحميل سريعًا، وهو أمر بالغ الأهمية لـ Core Web Vitals.
يجب أن تكون إعدادات الألوان مرنة بما يكفي لتغطية معظم متطلبات العلامات التجارية.
اختبارات القبول:

تحديث فوري للشعار: رفع شعار جديد من Admin Panel → يظهر الشعار الجديد في رأس Storefront فورًا بعد التحديث.
تغيير الألوان: تغيير اللون الأساسي للمتجر من Admin Panel → تتغير الألوان الرئيسية في Storefront ديناميكيًا.
عزل الإعدادات: إعدادات الهوية خاصة بالـ Tenant الحالي ولا تؤثر على أي Tenants آخرين.
صور محسّنة: الشعار والأيقونة يتم تحميلهما كصور محسّنة (WebP) مع Imgproxy.
اختبارات Pub/Sub: مراقبة Redis Pub/Sub تؤكد أن أحداث التحديث يتم إرسالها وتلقيها بشكل صحيح.

Super-#02 God Mode (Impersonation) ⚡🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

خاصية "God Mode" أو الانتحال تسمح للمشرفين العامين (Super Admins) بالوصول إلى واجهة Admin Panel لأي Tenant كما لو كانوا هم المشرفين على هذا Tenant، وذلك لأغراض الدعم، التتبع، أو الإدارة، مع الحفاظ على الأمان والمساءلة.

واجهة المستخدم (Super Admin UI):
النقطة الرئيسية: توفير نقطة دخول سهلة لخاصية الانتحال.
كيفية التنفيذ: إضافة زر "Impersonate" كإجراء (action) لكل Tenant في جدول نظرة عامة على Tenants في Super Admin UI (Super-#01). هذا الزر هو نقطة البداية لعملية الانتحال.
نقطة نهاية (API Endpoint) آمنة:
النقطة الرئيسية: معالجة طلب الانتحال بشكل آمن على الخادم.
كيفية التنفيذ: إنشاء API endpoint آمن في Backend خاص بـ Super Admin (مثال: POST /super-admin/impersonate/{tenantId}). يتطلب هذا الـ Endpoint مصادقة Super Admin ويتحقق من صلاحياته قبل معالجة الطلب.
توليد JWT خاص للانتحال:
النقطة الرئيسية: إصدار رمز مميز (Token) خاص يحدد جلسة الانتحال.
كيفية التنفيذ: عند استدعاء الـ Endpoint، يقوم @apex/auth module (أو خدمة مصادقة مخصصة) بتوليد JSON Web Token (JWT) جديد بخصائص فريدة:
sub (الموضوع): userId الخاص بالـ Super Admin الفعلي.
impersonating_as_tenant_id: tenantId الخاص بالـ Tenant المستهدف.
impersonating_as_user_id: userId وهمي يمثل Tenant Admin (أو معرف عام للـ Admin) داخل Tenant المستهدف.
roles: tenant_admin لضمان الصلاحيات اللازمة داخل لوحة تحكم Tenant.
iat, exp (توقيت الإصدار والانتهاء): لضمان أمان الـ Token وحمايته من هجمات إعادة التشغيل.
أمان التوقيع: يتم توقيعه بمفتاح سري مختلف عن مفاتيح JWT العادية إذا أمكن، أو بمفتاح JWT Master قوي لزيادة الأمان.
تخزين Cookie وإعادة توجيه:
النقطة الرئيسية: تسليم الـ JWT إلى المتصفح وتوجيه المستخدم.
كيفية التنفيذ: يتم إرسال هذا الـ JWT كـ httpOnly, secure, SameSite=Lax cookie إلى متصفح Super Admin. بعد ذلك، يتم إعادة توجيه المتصفح إلى لوحة تحكم Admin الخاصة بالـ Tenant المستهدف (مثال: https://tenant-x.apex.com/admin).
التحقق من الانتحال (Backend):
النقطة الرئيسية: التأكد من أن جميع طلبات الـ API خلال جلسة الانتحال يتم معالجتها بشكل صحيح وآمن.
كيفية التنفيذ: في جميع الـ API endpoints داخل Admin Panel، يتم فحص الـ JWT الوارد.
إذا كان يحتوي على impersonating_as_tenant_id، يتم استخدام هذا الـ tenantId بواسطة TenantIsolationMiddleware (Arch-Core-04).
يتم التأكيد أيضًا على أن الـ sub في الـ JWT هو Super Admin وليس Tenant Admin عادي، مما يضمن أن جلسة الانتحال مميزة عن جلسات تسجيل الدخول العادية.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع الإجراءات التي تتم أثناء جلسة الانتحال لأغراض المساءلة.
كيفية التنفيذ: يتم تعديل AuditLoggingInterceptor (Arch-S4) لتسجيل جميع الإجراءات التي تتم أثناء الانتحال في Global Audit Log (Super-#11). يجب أن يشير السجل بوضوح إلى actual_user_id (Super Admin الذي قام بالانتحال) و impersonated_tenant_id و impersonated_user_id.
إنهاء الانتحال:
النقطة الرئيسية: توفير طريقة آمنة وسهلة لإنهاء جلسة الانتحال.
كيفية التنفيذ: يتم توفير زر "Stop Impersonating" في واجهة Admin Panel للـ Tenant. هذا الزر يقوم بحذف الـ JWT الخاص بالانتحال من الكوكيز وإعادة توجيه Super Admin إلى لوحة تحكم Super Admin الخاصة به.
المعلومات البرمجية التقنية:

@apex/auth (Authentication Module): مسؤول عن توليد وتوقيع JWTs.
JWT (JSON Web Tokens): الآلية الأساسية للمصادقة وتخزين معلومات جلسة الانتحال.
S2 Isolation (Arch-Core-04): TenantIsolationMiddleware المستخدم لفرض نطاق Tenant.
Super-#11 (Global Audit Log): سجل التدقيق المركزي لتتبع الأنشطة.
Arch-S4 (AuditLoggingInterceptor): المكون المسؤول عن تسجيل الأحداث في سجل التدقيق.
ملاحظات:

الأمان هو الأولوية القصوى لخاصية "God Mode". يجب أن تكون عملية توليد وتخزين والتحقق من JWT صارمة للغاية.
يجب أن يكون impersonating_as_user_id معرفًا فريدًا يمثل دور المشرف داخل Tenant لضمان تطبيق سياسات RBAC الخاصة بالـ Tenant بشكل صحيح.
التوثيق الشامل لتدفق الانتحال ومراجعته الأمنية أمر حاسم.
يجب أن تكون المدة الزمنية لـ JWT الانتحال قصيرة لتقليل مخاطر الأمان.
اختبارات القبول:

نجاح الانتحال: Super Admin ينقر على "Impersonate Tenant X" → يتم إعادة توجيهه بنجاح إلى لوحة تحكم Admin الخاصة بـ Tenant X، مع رؤية البيانات والصلاحيات كما لو كان Tenant Admin.
تسجيل شامل في Audit Log: أي إجراء يتم (مثال: إنشاء منتج، حذف عميل) أثناء الانتحال يتم تسجيله في Global Audit Log (Super-#11) مع تفاصيل actual_user_id (Super Admin)، impersonated_tenant_id، والإجراء المحدد.
إنهاء الانتحال الفعال: النقر على "Stop Impersonating" يؤدي إلى العودة الفورية إلى لوحة تحكم Super Admin، ويتم مسح الـ JWT الخاص بالانتحال من الكوكيز.
أمان صارم: محاولة الوصول إلى لوحة تحكم Admin لـ Tenant باستخدام JWT عادي لا يحتوي على impersonating_as_tenant_id، أو باستخدام JWT منتحل تم التلاعب به، يفشل ويعود بـ 403 Forbidden.
توثيق إجراءات الأمان: توثيق مفصل لآلية الانتحال، وتدفق JWT، ونقاط التحقق الأمنية في docs/security/impersonation.md.
Super-#03 Kill Switch ⚡🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُعد خاصية "Kill Switch" آلية أمان حرجة تسمح للمشرفين العامين بتعليق خدمات أي Tenant على الفور في حالات الطوارئ (مثل النشاط الضار، تجاوز الدفع، أو انتهاكات السياسة)، مع ضمان انتشار التغيير بسرعة عبر النظام.

واجهة المستخدم (Super Admin UI):
النقطة الرئيسية: توفير طريقة واضحة وفعالة لتفعيل/إلغاء تفعيل Kill Switch.
كيفية التنفيذ: في Super Admin UI، يتم إضافة زر تبديل (toggle switch) مرئي بوضوح أو زر "Suspend/Activate" بجانب كل Tenant في جدول نظرة عامة على Tenants (Super-#01).
تحديث حالة Tenant في DB:
النقطة الرئيسية: تسجيل حالة التعليق في مصدر البيانات الرئيسي.
كيفية التنفيذ: عند تفعيل "Kill Switch"، يتم تحديث عمود status في جدول public.tenants إلى suspended عبر API endpoint آمن (مثال: PATCH /super-admin/tenants/{tenantId}/status).
إبطال Cache فوري:
النقطة الرئيسية: ضمان انتشار التغيير في الوقت الفعلي عبر جميع مكونات النظام.
كيفية التنفيذ: يتم إرسال حدث Redis Pub/Sub (Arch-Core-07) (مثال: tenant_status_updated:{tenantId}) لإبطال Cache الخاص بالـ Tenant فورًا في جميع مثيلات Storefront و Admin Panel. هذا يضمن انتشار التغيير بسرعة عبر النظام.
تطبيق Middleware للعزل:
النقطة الرئيسية: نقطة التحكم الرئيسية لاعتراض الطلبات للـ Tenants المعلقة.
كيفية التنفيذ: TenantIsolationMiddleware (Arch-Core-04) هو نقطة التحكم الرئيسية. في كل طلب وارد إلى أي خدمة تابعة لـ Tenant، يتحقق Middleware من حالة status للـ Tenant في قاعدة البيانات أو Cache. إذا كانت suspended، يتم اعتراض الطلب.
استجابة 503 Service Unavailable:
النقطة الرئيسية: إبلاغ المستخدمين والأنظمة الخارجية بأن الخدمة غير متاحة مؤقتًا.
كيفية التنفيذ: يقوم Middleware بإرجاع استجابة HTTP 503 Service Unavailable لجميع الطلبات الواردة إلى Storefront و Admin Panel الخاص بهذا Tenant. يمكن تخصيص استجابة 503 لتعرض صفحة صيانة مخصصة (مثال: maintenance.html) مع رسالة ودية.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع عمليات تفعيل/إلغاء تفعيل Kill Switch لأغراض المساءلة.
كيفية التنفيذ: يتم تسجيل جميع عمليات تفعيل/إلغاء تفعيل Kill Switch في Global Audit Log (Super-#11) مع تفاصيل المستخدم الذي قام بالإجراء والتاريخ والوقت.
اختبارات الصمود:
النقطة الرئيسية: التأكد من أن تفعيل Kill Switch لا يؤثر على أداء Tenants الآخرين.
كيفية التنفيذ: تتضمن اختبارات Load Testing سيناريوهات لتفعيل Kill Switch لـ Tenant واحد والتحقق من عدم تأثر Tenants الآخرين أو أدائهم.
المعلومات البرمجية التقنية:

S2 Middleware (Arch-Core-04): TenantIsolationMiddleware لفرض عزل Tenant والتحقق من حالته.
Redis Pub/Sub (Arch-Core-07): لنشر تحديثات حالة Tenant في الوقت الفعلي وإبطال Cache.
Super-#11 (Global Audit Log): سجل التدقيق المركزي لتوثيق جميع الإجراءات.
@apex/db: للتفاعل مع جدول public.tenants في قاعدة البيانات.
ملاحظات:

يجب أن تكون عملية إبطال Cache موثوقة وفورية لضمان تطبيق Kill Switch بسرعة.
صفحة الصيانة المخصصة تضمن تجربة مستخدم أفضل وتجنب إرباك العملاء.
التحكم في الوصول إلى هذه الميزة يجب أن يكون صارمًا ويقتصر على Super Admins فقط.
اختبارات القبول:

تعليق فوري للخدمة: Super Admin ينقر على "Suspend Tenant Y" → في غضون 5 ثوانٍ، أي محاولة للوصول إلى tenant-y.apex.com (سواء Storefront أو Admin Panel) تعود بـ HTTP 503 Service Unavailable، أو صفحة صيانة مخصصة.
تسجيل دقيق في Audit Log: يتم تسجيل حدث TENANT_SUSPENDED في Global Audit Log (Super-#11) مع تفاصيل Super Admin الذي قام بالإجراء.
إعادة التفعيل الفعال: Super Admin ينقر على "Activate Tenant Y" → في غضون 5 ثوانٍ، يعود tenant-y.apex.com للعمل بشكل طبيعي.
عزل كامل: تفعيل Kill Switch لـ Tenant واحد لا يؤثر على أداء أو توفر أي Tenants آخرين.
توثيق Kill Switch: توثيق إجراءات Kill Switch، بما في ذلك كيفية تفعيله، إلغاء تفعيله، وتأثيره في docs/operations/kill-switch.md.
Super-#04 Resource Quotas 🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُطبق حصص الموارد (Resource Quotas) للتحكم في استخدام الموارد من قبل كل Tenant بناءً على خطة اشتراكه، مما يضمن الاستخدام العادل للموارد، منع الإفراط في الاستخدام، وتحديد مستويات الخدمة.

جدول resource_quotas Schema:
النقطة الرئيسية: تصميم هيكل قاعدة بيانات لتحديد حدود الموارد لكل خطة.
كيفية التنفيذ: تصميم جدول resource_quotas في قاعدة البيانات يربط plan_id (من جدول الخطط) بحدود الموارد المختلفة. مثال للأعمدة: plan_id (FK), max_products (عدد صحيح), max_staff_users (عدد صحيح), storage_mb_limit (ميغابايت), max_api_calls_per_month (عدد صحيح).
نقاط تحقق في Backend (Guards/Services):
النقطة الرئيسية: فرض الحصص على مستوى API لمنع تجاوز الحدود.
كيفية التنفيذ:
max_products: يتم تطبيق Guard (مثال: ProductQuotaGuard) على API endpoint لإنشاء المنتجات. قبل إتمام عملية الإنشاء، يستعلم الـ Guard عن عدد المنتجات الحالية للـ Tenant من قاعدة البيانات ويقارنها بـ max_products الخاص بخطة Tenant.
max_staff_users: يتم تطبيق Guard مماثل على API endpoint لإضافة موظفين جدد.
storage_mb_limit: عند رفع الملفات إلى MinIO (Arch-Core-02)، يقوم الـ Backend Service بمعرفة حجم الملف الذي تم رفعه وإضافته إلى current_storage_mb للـ Tenant (يتم تحديث هذا الرقم بشكل دوري أو عند الرفع). يتم رفض الرفع إذا تجاوز الحد. يمكن أيضاً استخدام MinIO quota policies على مستوى الـ Bucket إذا كان كل Tenant له Bucket خاص به.
max_api_calls_per_month: يمكن تتبع هذا الحد باستخدام Redis (Arch-Core-07) كمخزن لعدادات شهرية لكل Tenant، ويتم تطبيق Guard (موسع من Arch-S6) يتحقق من هذا الحد.
رسائل خطأ واضحة:
النقطة الرئيسية: إبلاغ المستخدم بوضوح عند تجاوز الحصة المخصصة.
كيفية التنفيذ: عند تجاوز أي حصة، تقوم الـ API بإرجاع استجابة HTTP 403 Forbidden مع رسالة خطأ محددة وواضحة (مثال: {"message": "Quota exceeded: Maximum products (100) reached for your Basic plan."}).
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع محاولات تجاوز الحصص لأغراض المراجعة والأمان.
كيفية التنفيذ: يتم تسجيل جميع محاولات تجاوز الحصص (مع تفاصيل Tenant، نوع المورد، الحد الذي تم تجاوزه) في Global Audit Log (Super-#11) لغرض المراجعة والتحليل الأمني.
واجهة المستخدم في Admin Panel:
النقطة الرئيسية: توفير رؤية واضحة للمستخدمين حول استخدامهم للموارد.
كيفية التنفيذ: في Admin Panel، يتم عرض استخدام الموارد الحالي (مثال: "50/100 منتج") ورسائل تحذيرية للـ Tenant Admins عند الاقتراب من تجاوز الحصص.
المعلومات البرمجية التقنية:

@apex/db (Drizzle ORM): للتعامل مع resource_quotas و tenant_config في قاعدة البيانات.
MinIO (Arch-Core-02): لتخزين الملفات وتطبيق حدود التخزين.
Arch-S6 (Rate Limiting base): يوفر الأساس لفرض حدود على عدد استدعاءات API.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
ملاحظات:

يجب أن تكون رسائل الخطأ دقيقة ومفيدة لتوجيه المستخدمين.
يجب تصميم الـ Guards بعناية لتقليل الأداء العام للطلبات.
يمكن تطبيق نظام الإنذار (alerting system) لإبلاغ Super Admins عند تجاوز Tenant لحد معين بشكل متكرر.
اختبارات القبول:

فرض Quota المنتجات: Tenant على الخطة "Basic" (بحد أقصى 100 منتج) يحاول إنشاء المنتج رقم 101 → الـ API تعود بـ 403 Forbidden مع رسالة خطأ واضحة.
فرض Quota التخزين: Tenant يحاول رفع ملف يتجاوز به حد التخزين (مثال: 1GB) → عملية الرفع تفشل مع رسالة خطأ "Storage limit exceeded".
تسجيل التجاوزات: جميع محاولات تجاوز الحصص يتم تسجيلها في Global Audit Log (Super-#11).
عرض Quota في UI: Tenant Admin يمكنه رؤية استخدام موارده الحالية وحدودها في Admin Panel.
توثيق: توثيق شامل لجميع Quotas المطبقة لكل خطة اشتراك في docs/billing/quotas.md.
Super-#07 Feature Gating 🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُمكن خاصية Feature Gating من التحكم في إتاحة الميزات للمستخدمين بناءً على خطة اشتراكهم، مما يتيح تقديم مستويات خدمة مختلفة، اختبار الميزات الجديدة، وإدارة دورة حياة الميزات.

جدول feature_flags:
النقطة الرئيسية: تصميم هيكل قاعدة بيانات لتخزين حالة الميزات.
كيفية التنفيذ: تصميم جدول feature_flags في قاعدة البيانات لتخزين حالة الميزات. الأعمدة: feature_name (سلسلة فريدة), plan_id (FK إلى جدول الخطط), enabled (boolean). يمكن أن يكون هناك إدخال plan_id = NULL للميزات المتاحة لجميع الخطط أو الميزات العامة.
Caching الـ Feature Flags:
النقطة الرئيسية: تحقيق أقصى سرعة وصول لحالة الميزات.
كيفية التنفيذ: يتم تحميل جميع الـ Feature Flags عند بدء تشغيل التطبيق وتخزينها في Redis Cache (Arch-Core-07) لتحقيق أقصى سرعة وصول وتقليل حمل قاعدة البيانات.
تطبيق Feature Flags في Backend (NestJS):
النقطة الرئيسية: فرض التحكم في الوصول للميزات على مستوى API.
كيفية التنفيذ:
Custom Decorator و Guard: تطوير NestJS Custom Decorator (مثال: @FeatureFlag('ai_writer')) و Guard مخصص (مثال: FeatureFlagGuard).
يتم تطبيق الـ Decorator على Controllers أو Router methods. الـ Guard يقوم بالتحقق من:
tenantId الحالي (من TenantIsolationMiddleware - Arch-Core-04).
خطة الاشتراك الخاصة بهذا الـ Tenant.
حالة enabled للميزة المطلوبة من Redis Cache.
إذا لم يتم تفعيل الميزة للـ Tenant، يتم إرجاع HTTP 403 Forbidden.
تطبيق Feature Flags في Frontend (React):
النقطة الرئيسية: إخفاء أو تعطيل عناصر واجهة المستخدم للميزات غير المتاحة.
كيفية التنفيذ: عند تحميل Admin Panel (أو Storefront)، يتم جلب قائمة الـ Feature Flags المتاحة للـ Tenant الحالي من API endpoint (مثال: GET /api/tenant/feature-flags). يتم استخدام هذه القائمة ديناميكيًا لإخفاء أو تعطيل عناصر واجهة المستخدم (أزرار، أقسام، قوائم) التي لا تتوفر للـ Tenant، مما يمنع المستخدم من محاولة الوصول إلى ميزات غير متاحة.
إدارة من Super Admin UI:
النقطة الرئيسية: توفير واجهة مركزية لإدارة الميزات.
كيفية التنفيذ: في Super Admin Panel، توجد واجهة لـ Super Admins لتفعيل/تعطيل الميزات لخطط اشتراك معينة.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع تغييرات حالة الـ Feature Flags.
كيفية التنفيذ: يتم تسجيل جميع تغييرات حالة الـ Feature Flags في Global Audit Log (Super-#11).
المعلومات البرمجية التقنية:

Redis Cache (Arch-Core-07): لتخزين Feature Flags من أجل الوصول السريع.
S2 Isolation (Arch-Core-04): لتوفير سياق tenantId للـ Guards.
NestJS Decorators/Guards: لتطبيق منطق Feature Gating على مستوى Backend.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
@apex/db: للتعامل مع جدول feature_flags.
ملاحظات:

من الضروري تطبيق Feature Gating على كل من Frontend (لتحسين تجربة المستخدم) و Backend (للأمان ومنع الوصول غير المصرح به).
يجب أن تكون آلية Caching لـ Feature Flags سريعة ودقيقة لضمان تطبيق التغييرات فورًا.
يمكن استخدام Feature Gating أيضًا لاختبار A/B أو لإطلاق ميزات تدريجيًا لمجموعة محددة من المستخدمين.
اختبارات القبول:

إخفاء الميزات في UI: Super Admin يعطل ميزة "AI Writer" للخطة "Basic" → Tenant Admin على الخطة "Basic" لا يرى زر "Generate Description" أو قسم "AI Content" في Admin Panel.
حماية API Backend: Tenant Admin على الخطة "Basic" يحاول استدعاء API endpoint الخاص بـ "AI Writer" مباشرة → يتم رفض طلبه بـ 403 Forbidden من قبل FeatureFlagGuard.
تفعيل الميزات بشكل صحيح: Super Admin يفعل ميزة "AI Writer" للخطة "Pro" → Tenant Admin على الخطة "Pro" يرى زر "Generate Description" ويعمل بشكل صحيح.
تسجيل التغييرات: جميع تغييرات حالة الـ Feature Flags يتم تسجيلها في Global Audit Log (Super-#11).
أداء Caching: مراقبة Redis Cache تظهر cache hit ratio > 98% لعمليات جلب Feature Flags.
Super-#09 Dunning Management 🌐🛡️
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

نظام Dunning Management هو عملية إدارة الاشتراكات التي فشلت دفعاتها. يهدف إلى استرداد المدفوعات المستحقة بطريقة مؤتمتة ومنظمة، مع تقليل فقدان العملاء وتأثير ذلك على إيرادات الشركة.

Cron Job في BullMQ:
النقطة الرئيسية: جدول مراجعة دورية للاشتراكات.
كيفية التنفيذ: إنشاء مهمة BullMQ متكررة (مثال: dunning_check_job) يتم جدولتها للتشغيل يوميًا (مثال: في منتصف الليل بالتوقيت العالمي).
تحديد الاشتراكات الفاشلة:
النقطة الرئيسية: تحديد الاشتراكات التي تتطلب إجراء Dunning.
كيفية التنفيذ: تقوم هذه المهمة بالاستعلام في قاعدة البيانات (PostgreSQL) لتحديد جميع الاشتراكات النشطة التي فات موعد دفعها أو فشل دفعها (باستخدام معلومات من Stripe Webhooks أو نظام الفوترة الداخلي).
منطق إعادة المحاولة المكون من 3 مراحل:
النقطة الرئيسية: تطبيق تسلسل آلي للإجراءات لاسترداد المدفوعات.
كيفية التنفيذ:
اليوم 1 (فشل الدفع الأولي): إذا كان أول فشل دفع، يتم إرسال مهمة BullMQ لإرسال بريد إلكتروني تلقائي للعميل يبلغه بفشل الدفع ويطلب تحديث معلومات الدفع.
اليوم 3 (تذكير وتخفيض جزئي): إذا استمر فشل الدفع بعد 3 أيام، يتم إرسال مهمة BullMQ لإرسال بريد إلكتروني تذكيري ثانٍ. يمكن في هذه المرحلة، وبشكل اختياري، تفعيل تخفيض جزئي للميزات (مثال: تقليل max_products مؤقتًا) باستخدام نظام Feature Gating (Super-#07) أو Resource Quotas (Super-#04).
اليوم 7 (تعليق وإشعار نهائي): إذا استمر فشل الدفع بعد 7 أيام، يتم إرسال مهمة BullMQ لإرسال بريد إلكتروني نهائي للعميل يبلغه بتعليق حسابه. بعد ذلك، يتم استدعاء "Kill Switch" (Super-#03) لتعليق Tenant بالكامل.
إرسال البريد الإلكتروني:
النقطة الرئيسية: إدارة إرسال جميع الإشعارات المتعلقة بـ Dunning.
كيفية التنفيذ: يتم إرسال جميع رسائل البريد الإلكتروني المتعلقة بـ Dunning كمهام في emailQueue الخاصة بـ BullMQ (Arch-Core-07).
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع الإجراءات المتعلقة بـ Dunning لأغراض المساءلة.
كيفية التنفيذ: يتم تسجيل جميع الإجراءات المتعلقة بـ Dunning (فشل دفع، إرسال بريد إلكتروني، تخفيض ميزات، تعليق Tenant) في Global Audit Log (Super-#11).
تكوين مرن:
النقطة الرئيسية: السماح بالمرونة في تحديد قواعد Dunning.
كيفية التنفيذ: يجب أن تكون فترات الأيام ونوع الإجراءات قابلة للتكوين (configurable) ليتمكن Super Admin من تعديلها حسب الحاجة.
المعلومات البرمجية التقنية:

BullMQ (Arch-Core-07): لجدولة وتشغيل مهام Dunning وإرسال رسائل البريد الإلكتروني.
Mailpit (Arch-Core-02): (لبيئة التطوير) لاختبار وتصحيح أخطاء رسائل البريد الإلكتروني.
Super-#03 (Kill Switch): لتعليق Tenant بالكامل عند فشل الدفع المستمر.
Super-#07 (Feature Gating): لتطبيق تخفيض جزئي للميزات.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
@apex/db: للتفاعل مع جداول الاشتراكات والدفع في قاعدة البيانات.
ملاحظات:

يجب أن يكون منطق Dunning واضحًا للعملاء لتجنب أي سوء فهم أو إحباط.
التكامل مع نظام الفوترة (مثال: Stripe) أمر حيوي لتحديد حالات فشل الدفع بدقة.
يجب أن تكون رسائل البريد الإلكتروني واضحة، مهذبة، وتوفر تعليمات سهلة لمتابعة الدفع.
اختبارات القبول:

تسلسل إشعارات الديون: محاكاة فشل دفع لاشتراك → يتم إرسال بريد إلكتروني للعميل في اليوم الأول، ثم بريد إلكتروني آخر في اليوم الثالث (التحقق عبر Mailpit في بيئة التطوير).
تعليق تلقائي: بعد 7 أيام من فشل الدفع المستمر → يتم تعليق Tenant تلقائيًا باستخدام Kill Switch (Super-#03)، ويتم تسجيل هذا الحدث في Global Audit Log (Super-#11).
تطبيق التخفيض (إذا تم تكوينه): إذا تم تكوين تخفيض للميزات في اليوم 3، يتم التحقق من أن الميزات قد تم تخفيضها بالفعل (مثال: عدد المنتجات المرئية).
توثيق كامل: توثيق كامل لـ Dunning workflow، بما في ذلك التسلسل الزمني، المحفزات، والإجراءات لكل خطوة في docs/billing/dunning.md.
Super-#11 Global Audit Log ⚡🌐👁️
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

يوفر Global Audit Log (سجل التدقيق العالمي) رؤية مركزية وشاملة لجميع الأنشطة الهامة التي تحدث عبر النظام، مما يضمن المساءلة، يساعد في تتبع المشكلات، ويلبي متطلبات الامتثال.

واجهة المستخدم (Super Admin UI):
النقطة الرئيسية: توفير واجهة سهلة للوصول إلى سجل التدقيق.
كيفية التنفيذ: في Super Admin UI، يتم إنشاء صفحة مخصصة لعرض Global Audit Log، والتي يمكن الوصول إليها فقط بواسطة Super Admins.
مكون جدول تفاعلي:
النقطة الرئيسية: عرض سجلات التدقيق بطريقة منظمة وقابلة للبحث.
كيفية التنفيذ: استخدام مكتبة قوية للجدول (مثال: TanStack Table أو AG Grid) لإنشاء جدول تفاعلي يدعم:
البحث العام: حقل بحث عام للبحث في جميع الأعمدة (مثل action, user_id, resource_id).
الفلاتر المتقدمة: فلاتر لكل عمود تسمح بالبحث حسب action_type (مثال: PRODUCT_DELETED, TENANT_SUSPENDED), date_range (باستخدام DateTime pickers), tenant_id (من قائمة منسدلة), user_id، وحتى البحث داخل حقل changes_jsonb.
الفرز: فرز تصاعدي/تنازلي حسب timestamp وأي عمود آخر.
Pagination: لتقسيم كميات كبيرة من البيانات.
مصدر البيانات:
النقطة الرئيسية: جلب البيانات من جدول سجلات التدقيق.
كيفية التنفيذ: يتم جلب البيانات مباشرة من جدول audit_logs (Arch-S4). يمكن تحسين أداء الاستعلامات باستخدام Indexing مناسب على الأعمدة المستخدمة في البحث والفرز.
عرض التفاصيل:
النقطة الرئيسية: توفير عرض مفصل لكل سجل تدقيق.
كيفية التنفيذ: عند النقر على سجل مراجعة، يتم عرض نافذة منبثقة تعرض التفاصيل الكاملة للسجل، بما في ذلك الـ JSON الخاص بـ changes_jsonb بشكل منظم ومقروء.
عدم قابلية التغيير (Immutable):
النقطة الرئيسية: ضمان عدم إمكانية تعديل أو حذف سجلات التدقيق.
كيفية التنفيذ: يتم التأكيد في تصميم قاعدة البيانات على أن جدول audit_logs هو append-only. لا يمكن لأي عملية (حتى Super Admin) تعديل أو حذف السجلات.
تصدير البيانات:
النقطة الرئيسية: توفير إمكانية تصدير سجلات التدقيق.
كيفية التنفيذ: توفير زر "Export to CSV" الذي يقوم بـ:
تشغيل مهمة BullMQ (Arch-Core-07) في الخلفية لتصدير سجلات المراجعة (خاصة للكميات الكبيرة).
يتم إنشاء ملف CSV وتخزينه في MinIO (Arch-Core-02).
يتم تزويد Super Admin برابط تنزيل آمن ومؤقت للملف.
المراقبة:
النقطة الرئيسية: التأكد من أن أداء سجل التدقيق لا يؤثر على النظام.
كيفية التنفيذ: يتم مراقبة أداء استعلامات Audit Log لضمان عدم تأثيرها على أداء قاعدة البيانات.
المعلومات البرمجية التقنية:

PostgreSQL (Database, Arch-S4): لتخزين سجلات التدقيق وضمان خصائص عدم قابلية التغيير.
BullMQ (Arch-Core-07): لجدولة مهام تصدير سجلات التدقيق.
MinIO (Arch-Core-02): لتخزين ملفات CSV المصدرة.
TanStack Table (UI Component): لبناء الجدول التفاعلي في Super Admin UI.
ملاحظات:

ضمان عدم قابلية التغيير لسجلات التدقيق أمر بالغ الأهمية للمساءلة والأمان والامتثال.
الindexing الجيد على جدول audit_logs ضروري للحفاظ على أداء البحث والفلترة مع نمو حجم البيانات.
يجب أن تكون ميزة التصدير قادرة على التعامل مع كميات هائلة من البيانات بشكل فعال.
اختبارات القبول:

عرض شامل للسجلات: Super Admin يمكنه عرض جميع سجلات المراجعة من جميع Tenants في جدول واحد.
وظائف البحث والفلترة الدقيقة: استخدام جميع فلاتر البحث المتقدمة يعرض النتائج الصحيحة بدقة (مثال: فلترة حسب action: TENANT_SUSPENDED و date_range: past 30 days).
تصدير ناجح للبيانات الكبيرة: طلب تصدير 500,000 سجل مراجعة → يتم إنشاء ملف CSV وتوفير رابط تنزيل خلال 5 دقائق كحد أقصى.
عدم قابلية التغيير مؤكدة: اختبارات Pen-testing أو محاولات التعديل المباشر على قاعدة البيانات تفشل في تعديل أو حذف سجلات audit_logs.
أداء الجدول: تحميل وعرض 10,000 سجل مراجعة في أقل من 3 ثوانٍ.
Super-#12 Feature Flags UI 🌐
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُوفر واجهة إدارة Feature Flags للمشرفين العامين (Super Admins) القدرة على تفعيل أو تعطيل الميزات على مستوى النظام بالكامل، مما يتيح التحكم الديناميكي في سلوك التطبيق، إطلاق الميزات، أو تفعيل وضع الصيانة.

واجهة المستخدم (Super Admin UI):
النقطة الرئيسية: توفير لوحة تحكم مركزية وواضحة لإدارة Feature Flags.
كيفية التنفيذ: في Super Admin UI، يتم إنشاء صفحة مخصصة لإدارة الـ System-wide Feature Flags.
عرض Feature Flags:
النقطة الرئيسية: تقديم قائمة قابلة للتفاعل بجميع الـ Feature Flags.
كيفية التنفيذ: يتم عرض قائمة بجميع الـ System-wide Feature Flags المحددة مسبقًا (مثال: maintenance_mode, new_checkout_flow_beta, global_promo_banner). كل Flag يتم عرضه كزر تبديل (toggle switch) مع وصف موجز لوظيفته.
تحديث DB و Redis:
النقطة الرئيسية: ضمان حفظ التغييرات وتعميمها عبر النظام.
كيفية التنفيذ: عند قيام Super Admin بتغيير حالة Feature Flag (تفعيل/تعطيل):
يتم استدعاء API endpoint آمن في Backend (مثال: PATCH /super-admin/system-feature-flags/{flagName}).
يقوم Backend بتحديث حالة enabled لهذا الـ Feature Flag في جدول feature_flags في قاعدة البيانات (Super-#07).
الأهم من ذلك، يتم إرسال حدث Redis Pub/Sub (Arch-Core-07) (مثال: system_feature_flag_updated:{flagName}) لإبطال Cache الخاص بهذا الـ Feature Flag في جميع مثيلات التطبيق النشطة (API, Worker, Storefronts).
انتشار التغييرات في الوقت الفعلي:
النقطة الرئيسية: تطبيق تغييرات Feature Flag فورًا عبر النظام.
كيفية التنفيذ: جميع الخدمات (API Gateway, Storefronts, Workers) تستمع إلى هذه الأحداث. عند تلقي حدث تحديث، تقوم بإعادة جلب حالة الـ Feature Flag المعني وتحديث حالتها الداخلية، مما يضمن انتشار التغييرات في الوقت الفعلي عبر النظام.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع تغييرات حالة الـ Feature Flags.
كيفية التنفيذ: يتم تسجيل جميع تغييرات حالة الـ Feature Flags (مثال: SYSTEM_FLAG_MAINTENANCE_MODE_ENABLED) في Global Audit Log (Super-#11).
تكوين آمن:
النقطة الرئيسية: حماية هذه الميزة الهامة من الوصول غير المصرح به.
كيفية التنفيذ: يجب أن تكون هذه الميزة محمية بـ RBAC صارم لضمان أن Super Admins فقط يمكنهم تعديل هذه الـ Flags.
المعلومات البرمجية التقنية:

Redis Pub/Sub (Arch-Core-07): لنشر تحديثات Feature Flags في الوقت الفعلي.
S2 Isolation (Arch-Core-04): لتوفير سياق Tenant (إذا كانت هناك flags خاصة بـ tenant).
Super-#07 (underlying flag mechanism): تستخدم نفس الآلية الأساسية لتخزين الـ Feature Flags.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
@apex/db: للتعامل مع جدول feature_flags.
ملاحظات:

تُعد إمكانية تغيير سلوك النظام ديناميكيًا من خلال Feature Flags أداة قوية، ولكنها تتطلب حذرًا شديدًا.
يجب أن تكون عملية انتشار التغييرات سريعة وموثوقة لضمان اتساق سلوك التطبيق.
التوثيق الجيد لكل Feature Flag ضروري لفهم وظيفته وتأثيراته المحتملة.
اختبارات القبول:

تفعيل/تعطيل فوري: Super Admin يقوم بتغيير حالة maintenance_mode إلى ON → في غضون 5 ثوانٍ، جميع Storefronts تعرض صفحة الصيانة المخصصة.
تسجيل التغييرات: يتم تسجيل حدث SYSTEM_FLAG_MAINTENANCE_MODE_ENABLED في Global Audit Log (Super-#11) مع تفاصيل Super Admin الذي قام بالإجراء.

Arch-DB-10 Read/Write Splitting (CQRS) 🚀
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تقسيم القراءة/الكتابة (Read/Write Splitting)، وهو تطبيق لمبدأ CQRS (Command Query Responsibility Segregation)، يهدف إلى تحسين أداء قابلية التوسع لقاعدة البيانات عن طريق فصل أحمال عمل القراءة عن أحمال عمل الكتابة، وتوجيهها إلى قواعد بيانات محسّنة لكل غرض.

إعداد PostgreSQL Master-Replica:
النقطة الرئيسية: إنشاء بنية تحتية لقاعدة البيانات تدعم الفصل.
كيفية التنفيذ: تهيئة PostgreSQL للعمل بنظام Master-Replica Replication (مثال: Streaming Replication).
الـ Master هو الأساسي لعمليات الكتابة (INSERT, UPDATE, DELETE).
الـ Replica هي نسخة طبق الأصل للقراءة (SELECT).
يجب أن يكون الـ Replica في نفس المنطقة الجغرافية أو منطقة قريبة جدًا لتقليل Latency.
توجيه استعلامات Drizzle ORM:
النقطة الرئيسية: توجيه الاستعلامات بشكل ذكي إلى الخادم المناسب.
كيفية التنفيذ: تعديل Drizzle ORM Configuration في تطبيقات Backend (API, Worker) لتعريف connection strings مختلفة لـ Read و Write.
جميع استعلامات الكتابة (INSERT, UPDATE, DELETE) توجه إلى Primary DB.
جميع استعلامات القراءة البسيطة (SELECT) يمكن أن تظل موجهة إلى Primary لتحقيق الاتساق الفوري (Strong Consistency).
استعلامات التقارير/التحليلات الثقيلة: يتم تعديل الـ API endpoints الخاصة بـ Admin Analytics Dashboards والتقارير المعقدة لتوجه استعلاماتها إلى Read-Replica DB (حيث يمكن قبول الاتساق النهائي - Eventual Consistency).
Materialized Views (لـ Reporting):
النقطة الرئيسية: تحسين أداء التقارير المعقدة.
كيفية التنفيذ:
لتقارير Admin Panel الأكثر تعقيدًا أو التي تتطلب تجميعًا مكثفًا للبيانات، يتم إنشاء Materialized Views في Primary DB (مثال: monthly_sales_summary_mv, product_performance_mv).
يتم تحديث هذه الـ Materialized Views بشكل دوري (مثال: كل 5 دقائق أو كل ساعة) عبر مهمة BullMQ (Arch-Core-07) التي تستدعي REFRESH MATERIALIZED VIEW CONCURRENTLY.
الـ Admin Dashboards تستعلم هذه الـ Materialized Views (عبر Read-Replica) بدلاً من الجداول الأصلية.
مراقبة Replication Lag:
النقطة الرئيسية: التأكد من أن Replica متزامنة مع Master بشكل كافٍ.
كيفية التنفيذ: مراقبة pg_stat_replication في PostgreSQL لضمان أن تأخر Replication بين Primary و Replica يظل ضمن حدود مقبولة (مثال: أقل من 5 ثوانٍ). يمكن استخدام Prometheus/Grafana (أو أدوات مراقبة DB أخرى) لتتبع هذا الـ Metric.
اختبارات الأداء والضغط (Load Testing):
النقطة الرئيسية: تقييم تأثير وتقييم فعالية الفصل.
كيفية التنفيذ: إجراء اختبارات Load Testing باستخدام k6 (Launch-03) لتقييم تأثير Read/Write Splitting على أداء النظام الكلي.
المعلومات البرمجية التقنية:

PostgreSQL Replication (Master-Replica Setup): البنية التحتية الأساسية للفصل.
BullMQ (Arch-Core-07): لجدولة تحديث Materialized Views.
Drizzle ORM: لأداة توجيه الاستعلامات بين Primary و Replica.
Monitoring Tools (Prometheus/Grafana): لمراقبة صحة وأداء قواعد البيانات.
ملاحظات:

أحد التحديات الرئيسية هو إدارة الاتساق (Consistency). الاستعلامات التي تتطلب أحدث البيانات يجب أن توجه إلى Primary DB.
يجب أن يتم تصميم Materialized Views بعناية لتقليل تأثير تحديثها على Primary DB.
تكلفة إضافة Replicas وتكاليف الإدارة يجب أن تؤخذ في الاعتبار.
اختبارات القبول:

فصل حمل القراءة/الكتابة: أثناء تشغيل تقرير تحليلي ثقيل في Admin Dashboard (يستغرق عادةً 30 ثانية لاستعلام مباشر)، يظل زمن استجابة Checkout API أقل من 100ms، ويظل استخدام CPU لـ Primary DB أقل من 20%.
توجيه الاستعلامات مؤكد: مراقبة سجلات PostgreSQL و Traces من OpenTelemetry (Arch-Core-08) تؤكد أن استعلامات الكتابة (INSERT, UPDATE, DELETE) تذهب إلى Primary، بينما استعلامات التقارير الثقيلة تذهب إلى Read-Replica.
Materialized Views محدثة: Materialized Views يتم تحديثها بنجاح في الفترات المحددة، وبياناتها تتطابق مع بيانات Primary DB (مع الأخذ في الاعتبار فترة التأخر).
تأخر Replication منخفض: تأخر Replication بين Primary و Replica لا يتجاوز 5 ثوانٍ في الظروف العادية.
توثيق مفصل: توثيق كامل لإعداد Replication، وكيفية توجيه الاستعلامات، ومنطق Materialized Views في docs/database/read-write-splitting.md.
Super-#18 Database Snapshots 🛡️
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُوفر خاصية Database Snapshots القدرة على إنشاء نسخ احتياطية للبيانات في نقاط زمنية محددة، وهي ضرورية للتعافي من الكوارث، حماية البيانات، وتوفير مرونة لإعادة الحالة السابقة للـ Tenants.

واجهة المستخدم (Super Admin UI):

كيفية التنفيذ: في Super Admin UI، إضافة زر "Create Backup" أو "Backup Tenant" في صفحة تفاصيل كل Tenant. يمكن أيضاً توفير خيار "Backup All Tenants" لنسخ احتياطي شامل.
نقطة نهاية Backend آمنة:
النقطة الرئيسية: معالجة طلب النسخ الاحتياطي بشكل آمن على الخادم.
كيفية التنفيذ: إنشاء API endpoint آمن في Backend (مثال: POST /super-admin/tenants/{tenantId}/backup) يستقبل طلب Backup ويتطلب مصادقة Super Admin.
تشغيل مهمة BullMQ للنسخ الاحتياطي:
النقطة الرئيسية: تنفيذ عملية النسخ الاحتياطي الفعلية بشكل غير متزامن.
كيفية التنفيذ: الـ Backend يقوم بإرسال مهمة BullMQ (Arch-Core-07) (مثال: database_backup_job) إلى worker service. هذه المهمة تقوم بـ:
تشغيل أمر pg_dump من داخل حاوية PostgreSQL (أو حاوية مخصصة).
لـ Backup خاص بـ Tenant واحد: استخدام pg_dump -Fc --data-only --schema=tenant_{} db_name > /tmp/backup.sql لتصدير بيانات Schema محددة فقط.
ضغط ملف الـ dump (مثال: gzip /tmp/backup.sql).
التخزين في MinIO:
النقطة الرئيسية: تخزين ملفات النسخ الاحتياطي في مكان آمن.
كيفية التنفيذ: يتم رفع ملف الـ dump المضغوط إلى MinIO (Arch-Core-02) في مجلد مخصص للنسخ الاحتياطية (مثال: backups/tenant_{}/YYYYMMDD_HHMMSS.sql.gz).
سياسات دورة حياة MinIO:
النقطة الرئيسية: إدارة تخزين النسخ الاحتياطية بمرور الوقت.
كيفية التنفيذ: تكوين سياسات دورة حياة (Lifecycle Policies) في MinIO لحذف النسخ الاحتياطية القديمة تلقائيًا بعد فترة زمنية محددة (مثال: 30 يومًا)، أو نقلها إلى تخزين أرشيفي أقل تكلفة بعد فترة (مثال: 7 أيام).
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع عمليات النسخ الاحتياطي.
كيفية التنفيذ: يتم تسجيل كل عملية نسخ احتياطي (سواء ناجحة أو فاشلة) في Global Audit Log (Super-#11).

كيفية التنفيذ: يتم إعداد Runbook مفصل يوضح خطوات استعادة Tenant من نسخة احتياطية موجودة في MinIO.

النقطة الرئيسية: ضمان وجود نسخ احتياطية منتظمة.
كيفية التنفيذ: بالإضافة إلى النسخ الاحتياطية اليدوية، يتم تكوين cron job في BullMQ لإنشاء نسخ احتياطية تلقائية يومية لجميع Tenants.
المعلومات البرمجية التقنية:

كيفية التنفيذ:

يتم تخزين تفضيل اللغة في cookie (مثال: باستخدام next-intl أو next-i18next الذي يتعامل مع الكوكيز) لضمان استمراريته عبر الجلسات والزيارات اللاحقة للموقع.
دعم RTL (Right-to-Left):
النقطة الرئيسية: تكييف الواجهة للغات التي تُكتب من اليمين إلى اليسار.
كيفية التنفيذ:
تطبيق دعم RTL للغات مثل العربية (على سبيل المثال، عند اختيار اللغة العربية، يتم تعيين dir="rtl" attribute على عنصر <html> باستخدام useLayoutEffect أو useEffect في React).
تطبيق TailwindCSS RTL variants (مثال: rtl:text-right, rtl:ml-auto, rtl:mr-0) لضبط تخطيطات CSS تلقائيًا للغات RTL، مما يضمن ظهور الواجهة بشكل صحيح.

كيفية التنفيذ:

كيفية التنفيذ: يتم خدمة الصورة المحسّنة عبر Imgproxy (Arch-Core-09) لضمان تحسينها النهائي (WebP, resizing, compression) لأداء Storefront، بغض النظر عن جودة الصورة الأصلية.

كيفية التنفيذ: يتم تسجيل كل عملية تحسين صورة بواسطة AI في Global Audit Log (Super-#11) مع تفاصيل المستخدم، الصورة الأصلية والمحسّنة لغرض المراجعة والتكاليف.

Admin-#39 Fraud Detection 🛡️
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

نقطة التكامل:
النقطة الرئيسية: تحديد مكان دمج كشف الاحتيال في سير عمل الطلب.
كيفية التنفيذ: يتم دمج كشف الاحتيال في عملية إنشاء الطلب (Checkout - Store-#06) في Backend، مباشرة بعد تأكيد الطلب وقبل معالجة الدفع النهائي. هذا يضمن أن يتم فحص الطلبات قبل خصم الأموال.
إرسال مهمة BullMQ:
النقطة الرئيسية: معالجة كشف الاحتيال بشكل غير متزامن.
كيفية التنفيذ: عند إنشاء طلب جديد، يتم إرسال مهمة BullMQ (Arch-Core-07) (مثال: fraud_check_job) إلى worker service لضمان المعالجة غير المتزامنة. هذا يمنع إبطاء عملية Checkout.
استدعاء خدمة الاحتيال الخارجية:
النقطة الرئيسية: التفاعل مع خدمات كشف الاحتيال المتخصصة.
كيفية التنفيذ:
يقوم الـ Worker باستدعاء خدمة كشف الاحتيال الخارجية (مثال: Sift Science, ClearSale, أو حلول Google Cloud مثل reCAPTCHA Enterprise مع تحليل السلوك).
يتم إرسال بيانات الطلب ذات الصلة (عنوان الشحن، عنوان IP، معلومات الدفع المجهولة، تفاصيل المنتج، سلوك المستخدم).
يتم استخدام S7 Encryption (Arch-S7) لمفاتيح API الخاصة بخدمة الاحتيال المخزنة في قاعدة البيانات، مما يضمن أمانها.
تخزين النتائج:
النقطة الرئيسية: حفظ نتائج تقييم الاحتيال.
كيفية التنفيذ: تقوم خدمة الاحتيال بإرجاع risk_score (مثال: 0-100) و recommendation (مثال: "approve", "review", "reject"). يتم تخزين هذه النتائج في جدول order_fraud_scores في قاعدة البيانات (PostgreSQL) وربطها بالطلب.
تحديث حالة الطلب:
النقطة الرئيسية: اتخاذ إجراءات تلقائية بناءً على نتائج الاحتيال.
كيفية التنفيذ: بناءً على risk_score و recommendation، يمكن تحديث حالة الطلب تلقائيًا إلى "Pending Review" إذا كان risk_score مرتفعًا، أو "Rejected" إذا كان recommendation واضحًا بالرفض.
واجهة المستخدم:
النقطة الرئيسية: عرض نتائج الاحتيال للمسؤولين.
كيفية التنفيذ: في Admin Panel (Admin-#17)، يتم عرض risk_score و recommendation للطلب، مع إبراز الطلبات عالية المخاطر (مثال: شارة "High Risk" حمراء) للسماح للمسؤول بمراجعتها يدويًا.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع أحداث كشف الاحتيال.
كيفية التنفيذ: يتم تسجيل جميع نتائج كشف الاحتيال والإجراءات المتخذة (مثال: ORDER_FLAGGED_HIGH_RISK, ORDER_STATUS_UPDATED_BY_FRAUD_SYSTEM) في Global Audit Log (Super-#11).
المعلومات البرمجية التقنية:

Redis Queue (Arch-Core-07): لجدولة مهام كشف الاحتيال في الخلفية.
External Fraud API / Google Cloud reCAPTCHA Enterprise: خدمات كشف الاحتيال التي تُستخدم لتحليل الطلبات.
S7 Encryption (Arch-S7): لتشفير مفاتيح API الخاصة بخدمة الاحتيال.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
Admin-#17 (Order Management): السياق الذي يتم فيه عرض نتائج الاحتيال للمسؤولين.
PostgreSQL: (بالافتراض) لتخزين order_fraud_scores وربطها بالطلب.
ملاحظات:

Mkt-Ext-05 Webhook DLQ & Circuit Breaker 🛡️
تفاصيل الهيكل والنقاط الرئيسية والفرعية:

تُوفر خاصية Webhook DLQ & Circuit Breaker آلية قوية لإرسال Webhooks إلى تطبيقات الطرف الثالث، مع ضمان المرونة في مواجهة الفشل وحماية النظام من الأعطال الخارجية.

Webhook Worker مخصص:
النقطة الرئيسية: خدمة مخصصة لإرسال Webhooks.
كيفية التنفيذ: إنشاء خدمة worker مخصصة (BullMQ consumer - Arch-Core-07) لإرسال Webhooks إلى تطبيقات الطرف الثالث المسجلة (من Mkt-Ext-04). هذا يفصل منطق إرسال Webhook عن الـ API الرئيسي.
Circuit Breaker:
النقطة الرئيسية: حماية النظام من الأعطال المتكررة لتطبيقات الطرف الثالث.
كيفية التنفيذ:
دمج مكتبة opossum (أو بناء مخصص) لتطبيق نمط Circuit Breaker لكل Webhook endpoint (أي لكل تطبيق طرف ثالث).
يتم تكوين Circuit Breaker بـ failureThreshold (مثال: 5% من الطلبات تفشل)، resetTimeout (مثال: 60 ثانية قبل محاولة إعادة الإرسال)، timeout (مثال: 5 ثوانٍ لكل طلب Webhook).
إذا تجاوزت نسبة الفشل الحد المحدد، يفتح Circuit Breaker ("open state")، ويتم إيقاف إرسال Webhooks إلى هذا الـ Endpoint مؤقتًا، مما يمنع إغراق الطرف الثالث بطلبات فاشلة وحماية مواردنا.
Dead Letter Queue (DLQ):
النقطة الرئيسية: معالجة Webhooks الفاشلة أو التي تم إيقافها.
كيفية التنفيذ:
عندما يفتح Circuit Breaker، أو عندما تفشل مهمة Webhook بعد عدد معين من المحاولات (مثال: 3 محاولات)، يتم نقل مهمة Webhook إلى Dead Letter Queue (DLQ) مخصص في BullMQ.
يمكن مراجعة مهام DLQ من خلال BullMQ Dashboard، ويمكن إعادة معالجتها يدويًا لاحقًا أو إعداد آلية لإعادة المحاولة.
إشعارات:
النقطة الرئيسية: إبلاغ المسؤولين عند وجود مشكلة في إرسال Webhooks.
كيفية التنفيذ: عند فتح Circuit Breaker، يتم إرسال إشعار فوري (مثال: بريد إلكتروني، Slack) إلى Super Admin (أو فريق DevOps) لتنبيههم بوجود مشكلة في تكامل طرف ثالث محدد، مما يتيح التدخل السريع.
التسجيل في Audit Log:
النقطة الرئيسية: تتبع جميع أحداث Circuit Breaker و DLQ.
كيفية التنفيذ: يتم تسجيل جميع أحداث Circuit Breaker (فتح، إغلاق، نصف فتح) ومهام DLQ في Global Audit Log (Super-#11) مع تفاصيل التطبيق المتأثر.
إدارة Webhooks:
النقطة الرئيسية: السماح لـ Tenant Admin بتكوين Webhooks.
كيفية التنفيذ: في Admin Panel، يمكن لـ Tenant Admin تكوين Webhooks (URL, Events) التي يرغب في تلقيها، مما يمنحهم مرونة في تكاملاتهم.
المعلومات البرمجية التقنية:

BullMQ (Arch-Core-07): لجدولة وإدارة مهام Webhook و DLQ.
Redis (for BullMQ): لتخزين مهام BullMQ وحالاتها.
opossum (Circuit Breaker Library): لتطبيق نمط Circuit Breaker بشكل فعال.
Super-#11 (Global Audit Log): لسجل التدقيق المركزي.
Mkt-Ext-04 (Developer Portal & Apps): السياق الذي يتم فيه تسجيل تطبيقات الطرف الثالث وتحديد Webhook URLs.
NestJS (Backend Framework): (بالافتراض) لبناء Worker Service.
ملاحظات:

يجب أن تكون إعدادات Circuit Breaker قابلة للتكوين لتناسب متطلبات تطبيقات الطرف الثالث المختلفة ومستويات تحمل الفشل.
مراقبة DLQ أمر بالغ الأهمية لضمان عدم فقدان Webhooks وتوفر آلية للتعافي.
الإشعارات الفورية تساعد على حل المشكلات بسرعة وتقليل تأثير الأعطال.
اختبارات القبول:

فتح Circuit Breaker: محاكاة فشل 5 طلبات Webhook متتالية إلى تطبيق طرف ثالث (عن طريق تشغيل خدمة الطرف الثالث في وضع لا يستجيب) → Circuit Breaker يفتح، وتتوقف Webhooks عن الإرسال إلى هذا التطبيق.

إعادة المعالجة: Super Admin يمكنه مراجعة مهام DLQ وإعادة معالجتها يدويًا (بعد إصلاح خدمة الطرف الثالث).
إغلاق Circuit Breaker: بعد فترة resetTimeout، يحاول Circuit Breaker إرسال Webhook واحد (نصف فتح). إذا نجح، يتم إغلاق Circuit Breaker واستئناف الإرسال.
تسجيل الأحداث: يتم تسجيل جميع أحداث Circuit Breaker و DLQ في Global Audit Log (Super-#11).

التخزين المؤقت:
النقطة الرئيسية: تحسين أداء جلب التكوين.
كيفية التنفيذ: يمكن تخزين التكوين مؤقتًا على جانب العميل في تطبيق الهاتف المحمول (مثال: باستخدام AsyncStorage أو UserDefaults) لتقليل عدد الطلبات وتحسين الأداء، وعرض التكوين المخزن مؤقتًا أثناء جلب التكوين الجديد.
المعلومات البرمجية التقنية:

NativeWind 🌬️ (for dynamic styling in React Native): لتطبيق الأنماط ديناميكيًا في بيئة React Native.
S2 Isolation (Arch-Core-04): لفرض عزل البيانات على مستوى Tenant وضمان أن التكوين خاص بالـ Tenant.
Admin-#01 (Tenant Identity Settings): مصدر رئيسي لإعدادات هوية Tenant (الشعار، الألوان).
Super-#07 (Feature Gating): مصدر لحالة Feature Flags التي يمكن تفعيلها/تعطيلها في التطبيق.
NestJS (Backend Framework): (بالافتراض) لبناء API endpoint.
PostgreSQL: (بالافتراض) لتخزين tenant_config و feature_flags.
ملاحظات:

يجب أن يكون JSON Schema للتكوين موثقًا جيدًا لسهولة استخدامه في تطبيقات الهاتف المحمول.
آلية التخزين المؤقت ضرورية لتحسين تجربة المستخدم وتقليل اعتمادية التطبيق على الاتصال بالإنترنت.
تحديثات التكوين يجب أن تكون سريعة وموثوقة لضمان اتساق التجربة عبر جميع المنصات.
اختبارات القبول:

تحديث UI ديناميكي: تغيير الشعار أو اللون الأساسي لـ Tenant من Admin Panel (Admin-#01) → تطبيق الهاتف المحمول الخاص بالـ Tenant يعرض الشعار واللون الجديدين فورًا عند إعادة التشغيل أو تحديث البيانات داخل التطبيق.
تفعيل/تعطيل الميزات: Super Admin يغير حالة Feature Flag (Super-#07) (مثال: enable_ai_search) → تطبيق الهاتف المحمول يعكس هذا التغيير (مثال: يظهر زر "AI Search" أو يختفي) دون الحاجة لتحديث التطبيق من المتجر.
أداء Endpoint: استجابة GET /api/mobile/config تتم في أقل من 100ms.
تكوين موثق: توثيق JSON Schema المتوقع لـ Mobile UI Config في docs/mobile/sdui.md.
التحكم في التنقل: Super Admin يمكنه تعديل بنية التنقل في التطبيق (مثال: إضافة قسم جديد) عبر الـ JSON config.
