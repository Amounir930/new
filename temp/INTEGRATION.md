# دليل ربط القالب بنظام إدارة خلفي (Dashboard)

أحسنت! لقد أصبح لديك الآن واجهة أمامية (قالب) متكاملة وعصرية. الخطوة التالية لجعل هذا المتجر الإلكتروني حقيقيًا هي ربطه بنظام إدارة خلفي (لوحة تحكم أو Dashboard) لإدارة المحتوى الديناميكي مثل المنتجات والطلبات والعملاء.

هذا الدليل يشرح المفهوم والخطوات الأساسية للقيام بذلك.

## 1. فهم البنية الحالية (البيانات المؤقتة)

حاليًا، التطبيق يستخدم بيانات مؤقتة (mock data) مخزنة مباشرة في الكود، وتحديدًا في الملف: `src/lib/data.ts`.

هذا يعني أنه لتغيير منتج أو إضافة مقال، يجب تعديل هذا الملف يدويًا. هذا مناسب لمرحلة التصميم، ولكنه غير عملي لمتجر حقيقي.

## 2. ما هو النظام الخلفي (Backend) ولوحة التحكم (Dashboard)؟

*   **النظام الخلفي (Backend/API):** هو "عقل" التطبيق. إنه برنامج يعمل على خادم ويتصل بقاعدة بيانات (مثل Firestore, PostgreSQL, MongoDB). وظيفته هي توفير البيانات للواجهة الأمامية (القالب) عند الطلب، ومعالجة العمليات مثل إنشاء الطلبات وتسجيل المستخدمين.
*   **لوحة التحكم (Dashboard):** هي واجهة رسومية لك (كمدير للمتجر). تتيح لك إضافة منتجات جديدة، تعديل الأسعار، عرض الطلبات، وإدارة كل شيء دون الحاجة لكتابة أي كود. لوحة التحكم هذه تتواصل مع نفس النظام الخلفي.

**رسم توضيحي بسيط:**

```
[لوحة التحكم] <--> [النظام الخلفي (API)] <--> [قاعدة البيانات]
      ^
      |
      +---- [الواجهة الأمامية (القالب)]
```

## 3. خطوات الربط (نظرة عامة)

الربط يتطلب استبدال استدعاءات البيانات المحلية باستدعاءات للـ API الذي يوفره نظامك الخلفي.

### مثال: جلب المنتجات في صفحة المتجر

لنفترض أن لديك API يوفر قائمة المنتجات على الرابط: `https://api.your-store.com/products`

**الوضع الحالي (`src/app/shop/page.tsx`):**
يتم استيراد المنتجات مباشرة من ملف محلي.
```typescript
import { products } from '@/lib/data';

export default function ShopPage() {
  // ... يتم استخدام `products` هنا
}
```

**الوضع بعد الربط (باستخدام Next.js Server Components):**
يتم جلب المنتجات من الـ API عند تحميل الصفحة.
```typescript
// app/shop/page.tsx

async function getProducts() {
  const res = await fetch('https://api.your-store.com/products');
  if (!res.ok) {
    throw new Error('Failed to fetch products');
  }
  return res.json();
}

export default async function ShopPage() {
  const products = await getProducts();
  // ... يتم استخدام `products` هنا بنفس الطريقة
  // لكنها الآن بيانات حقيقية من قاعدة البيانات
}
```

### ما الذي يجب تغييره؟

يجب تطبيق هذا المبدأ على جميع أجزاء التطبيق التي تستخدم بيانات مؤقتة:

*   **المنتجات:** في جميع الصفحات (الرئيسية، المتجر، تفاصيل المنتج، إلخ).
*   **الفئات (Categories):** يجب جلبها من الـ API.
*   **مقالات المدونة (Blog Posts):** يجب جلبها من الـ API (أو من نظام إدارة محتوى مثل Contentful).
*   **عمليات المستخدم:**
    *   **تسجيل الدخول/إنشاء حساب:** يجب إرسال البيانات إلى الـ API للتحقق منها وإنشاء حسابات.
    *   **إضافة إلى السلة:** يمكن إدارتها من جانب العميل كما هي الآن، أو مزامنتها مع قاعدة البيانات عبر الـ API لحفظ السلات بين الأجهزة.
    *   **إتمام الطلب:** يجب إرسال تفاصيل الطلب إلى الـ API لتسجيله في قاعدة البيانات ومعالجته.
    *   **بيانات الحساب (الطلبات، العناوين):** يجب جلبها من الـ API الخاص بالمستخدم الذي قام بتسجيل الدخول.

## 4. اقتراحات إضافية

لتحسين التطبيق أكثر، يمكنك التفكير في إضافة الميزات التالية بعد ربط النظام الخلفي:

*   **تسجيل الدخول عبر الشبكات الاجتماعية (OAuth):** السماح للمستخدمين بتسجيل الدخول باستخدام حسابات Google أو Facebook.
*   **مقاطع فيديو للمنتجات:** إضافة حقل للفيديو في بيانات المنتج وعرضه في صفحة التفاصيل.
*   **بحث متقدم مع إكمال تلقائي:** تحسين شريط البحث ليقترح المنتجات أثناء الكتابة.
*   **دمج نظام إدارة محتوى (Headless CMS):** لربط المدونة بنظام مثل Strapi أو Contentful لتسهيل إدارة المقالات.

هذا الدليل يوفر لك خارطة طريق واضحة. ربط الواجهة الأمامية بنظام خلفي هو خطوة حاسمة ومثيرة في رحلة تطوير أي تطبيق ويب ناجح. بالتوفيق!
